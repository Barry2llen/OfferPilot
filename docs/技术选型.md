# OfferPilot 技术选型说明

## 1. 文档目标

本文档描述仓库当前阶段的实际技术选型，重点记录已经落地的后端、数据库、Agent、文档解析和测试体系。判断标准分为三类：

- 已落地：仓库里已经有代码和测试支撑。
- 已预留：目录、依赖或抽象已经存在，但能力尚未完整上线。
- 暂未引入：早期方案里提过，但当前代码库并未采用。

---

## 2. 当前项目定位

OfferPilot 目前是一个以 Python 为核心的求职助手后端原型。当前重点已经从纯服务骨架推进到可运行的后端基础设施和 Agent 执行底座：

- FastAPI 服务入口、生命周期管理和 OpenAPI 文档
- 简历文件上传、解析、预览和管理接口
- 模型供应商与模型选择 CRUD
- 基于 `SupervisorAgent` 的同步 AI 对话与 SSE 流式对话
- LangGraph checkpoint 持久化、工具调用事件输出和失败 interrupt/retry

当前仓库仍不是完整 Web 产品，也没有前端应用；它的定位是先把可扩展的 Agent 后端底座做扎实。

---

## 3. 已落地选型

### 3.1 运行时与包管理

- Python `>=3.13`
- `uv` 作为依赖管理和命令执行工具

选用原因：

- Python 生态与 LangChain、LangGraph、FastAPI、文档解析和 OCR 依赖兼容最好。
- `uv` 启动和依赖解析速度快，适合本地迭代和 CI 使用。

---

### 3.2 Web 框架：FastAPI

当前入口位于 `main.py`，使用 FastAPI 创建应用，并通过 lifespan 完成配置加载、数据库初始化、建表、checkpoint 初始化和 `SupervisorAgent` 装配。

当前状态：

- 已落地应用入口和生命周期管理
- 已注册 `resumes`、`model-config`、`ai` 三组 API tag
- 已提供 `/resumes` 简历文件管理接口
- 已提供 `/model-providers` 与 `/model-selections` 模型配置 CRUD
- 已提供 `/ai/chat` 同步对话和 `/ai/chat/stream` SSE 流式对话
- 路由层已补充 `summary`、`description`、`response_description` 和主要错误响应说明

选用原因：

- 与 Python Agent 生态衔接自然
- 异步支持和类型提示体验较好
- 自动生成 OpenAPI，适合当前阶段直接通过 Swagger 联调

---

### 3.3 配置系统：Pydantic v2 + ruamel.yaml

当前配置由 `schemas/config/` 负责，`load_config()` 从 `config.yaml` 读取并解析为强类型对象。

当前状态：

- 已支持根级 `database` 配置
- 已兼容旧版 `offer_pilot.database` 与 `offer_pilot.web_search` 结构
- 已支持 SQLite 与 PostgreSQL 两种数据库配置
- 已支持 `resume_upload_dir`、`debug`、`model_call_retry_attempts`
- 已支持 `web_search` 与 `exa_api_key`
- 已提供 `config.example.yaml`

选用原因：

- Pydantic v2 适合做强类型配置建模和默认值约束
- `ruamel.yaml` 保留 YAML 结构能力较好，便于后续扩展和示例维护

---

### 3.4 数据库层：SQLAlchemy 2.x

当前数据库访问基于 SQLAlchemy 2.x ORM，而不是早期方案中提到的 SQLModel。

当前状态：

- `db/engine/manager.py` 提供同步/异步 URL 构建、惰性初始化、会话工厂、连接检查与资源释放
- `db/models/` 已定义 `tb_resume`、`tb_chat`、`tb_model_provider`、`tb_model_selection`、LangGraph checkpoint 相关表
- `db/repositories/` 已提供简历、模型配置和 checkpoint 的数据访问封装
- 启动时通过 ORM metadata 自动建表
- `sql/tables.sql` 保留数据库结构快照，并要求 SQLite/PostgreSQL 兼容

选用原因：

- SQLAlchemy 2.x 表达力和可控性更强，适合做基础设施层
- 当前已经需要清晰的 ORM、Repository、Service 分层
- 对 SQLite 与 PostgreSQL 双兼容更直接

---

### 3.5 数据库类型：SQLite 默认，PostgreSQL 可选

当前默认数据库是 SQLite，配置路径为 `./data/offer_pilot.db`；同时保留 PostgreSQL 配置模型与可选测试。

当前状态：

- SQLite 为默认运行模式
- PostgreSQL 配置、URL 构建与连接检查已实现
- 同步 SQLite 使用 `sqlite`，异步 SQLite 使用 `sqlite+aiosqlite`
- PostgreSQL 使用 `psycopg`，测试依赖 `TEST_POSTGRES_*` 环境变量

选用原因：

- SQLite 启动门槛低，适合本地开发与原型验证
- PostgreSQL 作为后续部署或多环境运行的扩展方向，避免现在就把数据层锁死在单一实现

---

### 3.6 Checkpoint：数据库持久化 LangGraph 状态

当前 `agent/checkpointers/database.py` 提供 `DatabaseCheckpointer`，用于把 LangGraph checkpoint 写入当前配置的关系型数据库。

当前状态：

- 已支持同步 checkpoint 读写
- 已支持异步 checkpoint 读写
- 已覆盖 writes、复制删除、prune 和 round trip 测试
- `/ai/chat` 与 `/ai/chat/stream` 通过 `thread_id` 持久化会话状态
- 流式失败重试依赖同一 `thread_id` 恢复 LangGraph interrupt

选用原因：

- Agent 会话不能只停留在内存中
- LangGraph interrupt/retry 需要 checkpoint 支撑
- 复用关系型数据库能减少当前阶段的基础设施复杂度

---

### 3.7 Agent 编排：LangChain + LangGraph

当前 Agent 基础能力落在 `agent/` 目录，核心图实现是 `agent/graphs/model_call.py` 中的 `ModelCallGraph`，对外服务使用 `agent/agents/supervisor/`。

当前状态：

- 已支持模型调用节点
- 已支持工具调用节点
- 已支持模型输出 `tool_calls` 后回流工具执行
- 已支持多工具并发执行、工具错误隔离和缺失工具错误消息
- 已支持模型调用失败后的内部重试
- 已支持 LangGraph interrupt，并可通过 `Command(resume=...)` 恢复
- Agent 基础状态由 `TypedDict` 定义

LangChain 当前承担：

- 统一聊天模型初始化
- 工具绑定
- 消息对象与工具调用协议复用

LangGraph 当前承担：

- 状态图定义
- `model -> tool -> model` 闭环编排
- 条件分支与结束控制
- checkpoint、interrupt 和 resume

选用原因：

- 当前项目不是简单的单次 LLM 调用，而是需要保留工作流、工具调用、中断恢复和多节点扩展点
- LangGraph 比手写循环更适合后续增加分支、人工介入和复杂状态

---

### 3.8 模型接入抽象：多提供商统一封装

当前已经落地“模型提供商 + 模型选择”两层抽象：

- `ModelProvider`：描述供应商、配置名称、`base_url`、`api_key`
- `ModelSelection`：描述实际使用的模型，以及是否支持图片输入

已支持的提供商类型：

- OpenAI
- Google
- Anthropic
- OpenAI Compatible

当前状态：

- `/model-providers` 提供模型供应商 CRUD
- `/model-selections` 提供模型选择 CRUD
- API 响应只返回 `has_api_key`，不回显 API Key 明文
- `agent/models/chat.py` 通过 LangChain `init_chat_model()` 装载模型
- `OpenAI Compatible` 会映射为 OpenAI 协议并要求 `base_url` 与 `api_key`
- `Google` 会映射为 `google_genai`

选用原因：

- 避免把具体模型厂商硬编码在业务逻辑里
- 便于后续支持不同工作流绑定不同模型
- 兼容官方接口和自建兼容接口

---

### 3.9 工具系统：Exa Web Search + MCP 预留

当前工具入口位于 `agent/tools/`，由 `get_all_tools(config)` 注入 `SupervisorAgent`。

当前状态：

- 配置了 `exa_api_key` 时启用 Exa 工具
- 当前 Exa 工具包括 `web_search_exa`、`web_fetch_exa`、`find_similar_exa`
- 缺少 Exa Key 时默认禁用 Web 搜索工具并记录 warning
- MCP Web Search fallback 已预留，但默认不启用，避免测试或启动阶段误连远程服务
- SSE 流式接口会输出 `tool_start`、`tool_end`、`tool_error` 事件

选用原因：

- Web 搜索属于 Agent 的外部信息获取能力，应通过工具注入而不是写死在图节点里
- Exa 提供搜索、内容获取和相似页面能力，覆盖当前通用检索需求
- MCP 保留后续扩展空间，但不能影响默认本地运行稳定性

---

### 3.10 简历文件解析与预览

当前简历能力已经不是占位状态，已通过 `api/routes/resume.py`、`services/resume_service.py`、`schemas/resume_document.py` 和 `utils/document_parser.py` 落地。

当前状态：

- 支持上传、替换、列出、详情、删除和原文件预览
- 支持 PDF、DOCX、PNG、JPG、JPEG
- 不支持旧版 `.doc`
- PDF 文本提取与渲染使用 PyMuPDF
- DOCX 文本提取使用 ZIP + XML 解析
- DOCX 预览图片转换使用 Aspose.Words
- 图片 OCR 和 PDF OCR 使用 `rapidocr-onnxruntime`
- 上传文件保存在 `resume_upload_dir`

选用原因：

- PyMuPDF 覆盖 PDF 文本提取和页面渲染，适合简历预览
- 直接解析 DOCX XML 可以避免简单文本提取依赖过重
- Aspose.Words 用于 DOCX 到图片预览，比自行处理 Word 布局更可靠
- RapidOCR 适合本地 OCR 原型验证，不依赖外部 OCR 服务

---

### 3.11 日志与基础工具

当前日志通过 `loguru` 封装在 `utils/logger.py` 中；通用流事件处理位于 `utils/stream.py`。

选用原因：

- `loguru` 比标准库 logging 更快落地
- 输出格式和调用体验适合当前阶段调试
- 流事件工具为 LangChain/LangGraph 事件处理提供公共抽象

---

### 3.12 测试框架：pytest

当前测试集中在 `tests/unit/`，覆盖范围包括：

- 配置解析
- 数据库 URL、连接检查、建表和 PostgreSQL 可选集成
- LangGraph 数据库 checkpointer
- ORM、Repository、Service 层
- 简历文件解析、预览和 API
- 模型配置 API 和 OpenAPI 文档
- AI 同步/流式接口、SSE 工具事件、interrupt/retry
- Agent 图中的工具调用、并发、错误隔离、模型重试和异常逻辑
- 模型装载错误封装

选用原因：

- 与当前 Python 技术栈契合
- 便于围绕 Agent 节点、数据库兼容性和接口文档写细粒度测试
- 能在没有真实外部模型和远程工具服务时验证核心行为

---

## 4. 已预留但未完全落地

### 4.1 业务型 Agent 工作流

`agent/agents/resume_extractor/`、`agent/agents/resume_advice/`、`agent/agents/interview_prepare/` 和 `agent/workflows/resume_extract/` 已存在，但当前对外 API 使用的是 `SupervisorAgent`。

结论：

- Agent 基础图、模型调用、工具调用和 checkpoint 已落地
- 具体的简历优化、面试准备等业务工作流仍需要继续产品化和接入 API

---

### 4.2 RAG 与向量检索

当前已经有 Web 搜索工具能力，但没有落地独立向量库、Embedding 存储或长期知识库检索链路。

结论：

- 外部信息获取能力已通过工具预留
- ChromaDB、向量索引和完整 RAG pipeline 尚未引入

---

## 5. 暂未引入或不应写成“已采用”的方案

以下内容出现在旧版设想中，但当前仓库并未真正采用：

- Vue 3 + TypeScript + Element Plus 前端
- ChromaDB 向量库
- 已完成的 RAG 检索链路
- 完整的 JD 分析、简历优化、模拟面试、投递记录管理能力
- SQLModel
- Celery、Redis、消息队列或后台任务系统
- Alembic 数据库迁移体系

这些内容可以继续作为后续规划讨论，但不应在当前技术文档里表述为既成事实。

---

## 6. 当前架构结论

截至目前，OfferPilot 的实际技术底座可以概括为：

- Python 3.13
- uv
- FastAPI
- Pydantic v2
- ruamel.yaml
- SQLAlchemy 2.x
- SQLite 默认 + PostgreSQL 可选
- LangChain
- LangGraph
- 数据库持久化 checkpoint
- Exa Web Search 工具
- PyMuPDF
- Aspose.Words
- RapidOCR ONNX Runtime
- Loguru
- Pytest

项目当前最清晰的方向是“可扩展的 Agent 后端底座”：用 FastAPI 暴露稳定 API，用 SQLAlchemy 管理业务数据和 checkpoint，用 LangGraph 编排模型与工具调用，再逐步补齐更具体的求职业务工作流、RAG 和前端界面。
