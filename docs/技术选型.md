# OfferPilot 技术选型说明

## 1. 文档目标

本文档描述 OfferPilot 当前单仓库的实际技术选型，覆盖后端、前端、Electron 桌面壳、数据库、Agent、文档解析、测试与打包体系。判断标准分为三类：

- 已落地：仓库里已经有代码、配置或测试支撑。
- 已预留：目录、依赖或抽象已经存在，但能力尚未完整上线。
- 暂未引入：早期方案里提过，或常见于同类产品，但当前代码库并未采用。

---

## 2. 当前项目定位

OfferPilot 当前是一个本地优先的 AI 求职助手单仓库项目，由三个子项目组成：

- `backend/`：FastAPI 后端，提供简历文件管理、模型配置、AI 对话、Agent 编排、数据库与 checkpoint。
- `frontend/`：Next.js 前端，提供聊天、简历管理、模型供应商配置和模型选择配置 UI。
- `electron/`：Electron 桌面壳，负责本地启动后端与前端服务，并构建 Windows 桌面安装包。

当前已经落地的产品能力包括：

- 简历文件上传、解析、预览、替换、查询和删除。
- 模型供应商与模型选择 CRUD。
- 基于 `SupervisorAgent` 的同步 AI 对话与 SSE 流式对话。
- LangGraph checkpoint 持久化、工具调用事件输出和失败 interrupt/retry。
- Next.js Web UI 与 Electron 桌面启动/打包链路。

项目现阶段重点是把“本地可运行的 AI 求职助手”打通：后端提供稳定 API 和 Agent 底座，前端提供可用操作界面，Electron 负责桌面化和本地进程托管。

## 3. 已落地选型

### 3.1 单仓库结构：三项目独立依赖

当前仓库把原先独立的后端、前端和 Electron 项目合并到一个根目录，但没有引入统一 workspace：

- 后端依赖由 `backend/pyproject.toml` 和 `backend/uv.lock` 管理。
- 前端依赖由 `frontend/package.json` 和 `frontend/package-lock.json` 管理。
- Electron 依赖由 `electron/package.json` 和 `electron/package-lock.json` 管理。

选用原因：

- 合并仓库后便于跨端同步接口、类型和打包约定。
- 保留独立依赖能降低迁移风险，不需要立即重构成 npm workspace 或 uv workspace。
- Electron 打包脚本已经能从相邻 `backend/` 和 `frontend/` 获取产物。

---

### 3.2 后端运行时与包管理：Python 3.13 + uv

后端目标 Python 版本为 `>=3.13`，统一使用 `uv` 管理依赖和执行命令。

选用原因：

- Python 生态与 FastAPI、LangChain、LangGraph、文档解析和 OCR 依赖兼容度高。
- `uv` 依赖解析和命令启动速度快，适合本地迭代、测试和 PyInstaller 打包前置步骤。

---

### 3.3 Web API：FastAPI

后端入口位于 `backend/main.py`，使用 FastAPI 创建应用，并通过 lifespan 完成配置加载、数据库初始化、建表、checkpoint 初始化和 `SupervisorAgent` 装配。

当前状态：

- 已注册 `resumes`、`model-config`、`ai` 三组 API tag。
- 已提供 `/resumes` 简历文件管理接口。
- 已提供 `/model-providers` 与 `/model-selections` 模型配置 CRUD。
- 已提供 `/ai/chat` 同步对话和 `/ai/chat/stream` SSE 流式对话。
- 路由层已补充 `summary`、`description`、`response_description` 和主要错误响应说明。

选用原因：

- 与 Python Agent 生态衔接自然。
- 异步支持和类型提示体验较好。
- 自动生成 OpenAPI，适合前后端联调和前端类型同步。

---

### 3.4 配置系统：Pydantic v2 + ruamel.yaml

后端配置由 `backend/schemas/config/` 负责，`load_config()` 从 `config.yaml` 读取并解析为强类型对象。

当前状态：

- 支持根级 `database` 配置。
- 兼容旧版 `offer_pilot.database` 与 `offer_pilot.web_search` 结构。
- 支持 SQLite 与 PostgreSQL 两种数据库配置。
- 支持 `resume_upload_dir`、`debug`、`model_call_retry_attempts`。
- 支持 `web_search` 与 `exa_api_key`。
- 提供 `backend/config.example.yaml`。

选用原因：

- Pydantic v2 适合做强类型配置建模和默认值约束。
- `ruamel.yaml` 保留 YAML 结构能力较好，便于维护配置示例。

---

### 3.5 数据库层：SQLAlchemy 2.x

当前数据库访问基于 SQLAlchemy 2.x ORM，而不是早期方案中提到的 SQLModel。

当前状态：

- `db/engine/manager.py` 提供同步/异步 URL 构建、惰性初始化、会话工厂、连接检查与资源释放。
- `db/models/` 已定义 `tb_resume`、`tb_chat`、`tb_model_provider`、`tb_model_selection`、LangGraph checkpoint 相关表。
- `db/repositories/` 已提供简历、模型配置和 checkpoint 的数据访问封装。
- 启动时通过 ORM metadata 自动建表。
- `sql/tables.sql` 保留数据库结构快照，并要求 SQLite/PostgreSQL 兼容。

选用原因：

- SQLAlchemy 2.x 表达力和可控性更强，适合做基础设施层。
- 当前已经需要清晰的 ORM、Repository、Service 分层。
- 对 SQLite 与 PostgreSQL 双兼容更直接。

---

### 3.6 数据库类型：SQLite 默认，PostgreSQL 可选

当前默认数据库是 SQLite，配置路径为 `./data/offer_pilot.db`；同时保留 PostgreSQL 配置模型与可选测试。

当前状态：

- SQLite 为默认运行模式。
- PostgreSQL 配置、URL 构建与连接检查已实现。
- 同步 SQLite 使用 `sqlite`，异步 SQLite 使用 `sqlite+aiosqlite`。
- PostgreSQL 使用 `psycopg`，测试依赖 `TEST_POSTGRES_*` 环境变量。
- Electron 打包后会在 Electron `userData` 下创建后端运行时配置和 SQLite 数据目录。

选用原因：

- SQLite 启动门槛低，适合本地开发和桌面应用。
- PostgreSQL 作为后续部署或多环境运行的扩展方向，避免数据层锁死在单一实现。

---

### 3.7 Checkpoint：数据库持久化 LangGraph 状态

`backend/agent/checkpointers/database.py` 提供 `DatabaseCheckpointer`，用于把 LangGraph checkpoint 写入当前配置的关系型数据库。

当前状态：

- 支持同步 checkpoint 读写。
- 支持异步 checkpoint 读写。
- 已覆盖 writes、复制删除、prune 和 round trip 测试。
- `/ai/chat` 与 `/ai/chat/stream` 通过 `thread_id` 持久化会话状态。
- 流式失败重试依赖同一 `thread_id` 恢复 LangGraph interrupt。

选用原因：

- Agent 会话不能只停留在内存中。
- LangGraph interrupt/retry 需要 checkpoint 支撑。
- 复用关系型数据库能减少当前阶段的基础设施复杂度。

---

### 3.8 Agent 编排：LangChain + LangGraph

Agent 基础能力位于 `backend/agent/`，核心图实现是 `agent/graphs/model_call.py` 中的 `ModelCallGraph`，对外服务使用 `agent/agents/supervisor/`。

当前状态：

- 支持模型调用节点和工具调用节点。
- 支持模型输出 `tool_calls` 后回流工具执行。
- 支持多工具并发执行、工具错误隔离和缺失工具错误消息。
- 支持模型调用失败后的内部重试。
- 支持 LangGraph interrupt，并可通过 `Command(resume=...)` 恢复。
- Agent 基础状态由 `TypedDict` 定义。

LangChain 当前承担：

- 统一聊天模型初始化。
- 工具绑定。
- 消息对象与工具调用协议复用。

LangGraph 当前承担：

- 状态图定义。
- `model -> tool -> model` 闭环编排。
- 条件分支与结束控制。
- checkpoint、interrupt 和 resume。

选用原因：

- 当前项目不是简单的单次 LLM 调用，而是需要保留工作流、工具调用、中断恢复和多节点扩展点。
- LangGraph 比手写循环更适合后续增加分支、人工介入和复杂状态。

---

### 3.9 模型接入抽象：多提供商统一封装

当前已经落地“模型供应商 + 模型选择”两层抽象：

- `ModelProvider`：描述供应商、配置名称、`base_url`、`api_key`。
- `ModelSelection`：描述实际使用的模型，以及是否支持图片输入。

当前支持的提供商类型：

- OpenAI
- Google
- Anthropic
- DeepSeek
- OpenAI Compatible

当前状态：

- `/model-providers` 提供模型供应商 CRUD。
- `/model-selections` 提供模型选择 CRUD。
- API 响应只返回 `has_api_key`，不回显 API Key 明文。
- `agent/models/chat.py` 通过 LangChain `init_chat_model()` 装载模型。
- `OpenAI Compatible` 会映射为 OpenAI 协议并要求 `base_url` 与 `api_key`。
- `Google` 会映射为 `google_genai`。

选用原因：

- 避免把具体模型厂商硬编码在业务逻辑里。
- 便于后续支持不同工作流绑定不同模型。
- 兼容官方接口和自建兼容接口。

---

### 3.10 工具系统：Exa Web Search + MCP 预留

工具入口位于 `backend/agent/tools/`，由 `get_all_tools(config)` 注入 `SupervisorAgent`。

当前状态：

- 配置了 `exa_api_key` 时启用 Exa 工具。
- 当前 Exa 工具包括 `web_search_exa`、`web_fetch_exa`、`find_similar_exa`。
- 缺少 Exa Key 时默认禁用 Web 搜索工具并记录 warning。
- MCP Web Search fallback 已预留，但默认不启用，避免测试或启动阶段误连远程服务。
- SSE 流式接口会输出 `tool_start`、`tool_end`、`tool_error` 事件。

选用原因：

- Web 搜索属于 Agent 的外部信息获取能力，应通过工具注入而不是写死在图节点里。
- Exa 提供搜索、内容获取和相似页面能力，覆盖当前通用检索需求。
- MCP 保留后续扩展空间，但不能影响默认本地运行稳定性。

---

### 3.11 简历文件解析与预览

简历能力已通过 `backend/api/routes/resume.py`、`backend/services/resume_service.py`、`backend/schemas/resume_document.py` 和 `backend/utils/document_parser.py` 落地。

当前状态：

- 支持上传、替换、列出、详情、删除和原文件预览。
- 上传和替换接口使用 SSE 流式返回解析进度，事件包括 `resume`、`progress`、`model_error`、`final`、`error`。
- 解析结果通过 `tb_resume_extraction` 持久化，列表和详情接口会返回解析状态、摘要、章节和事实数量。
- 支持 PDF、DOCX、PNG、JPG、JPEG。
- 不支持旧版 `.doc`。
- PDF 文本提取与渲染使用 PyMuPDF。
- DOCX 文本提取使用 ZIP + XML 解析。
- DOCX 预览图片转换使用 Aspose.Words。
- 图片 OCR 和 PDF OCR 使用 `rapidocr-onnxruntime`。
- 上传文件保存在 `resume_upload_dir`。

选用原因：

- PyMuPDF 覆盖 PDF 文本提取和页面渲染，适合简历预览。
- 直接解析 DOCX XML 可以避免简单文本提取依赖过重。
- Aspose.Words 用于 DOCX 到图片预览，比自行处理 Word 布局更可靠。
- RapidOCR 适合本地 OCR 原型验证，不依赖外部 OCR 服务。

---

### 3.12 前端框架：Next.js 16 App Router + React 19

前端位于 `frontend/`，使用 Next.js `16.2.3`、React `19.2.4`、TypeScript strict 和 App Router。

当前状态：

- 路由入口位于 `frontend/app/`，没有 `src/` 或 `pages/` 目录。
- `app/layout.tsx` 负责 metadata、全局 CSS、`AppProvider`、`ToastProvider` 和 `AppShell`。
- `app/page.tsx` 是 AI 对话页，使用 SSE 流式聊天。
- `app/resumes/` 提供简历列表、上传、详情、预览、替换和删除。
- `app/settings/providers/` 与 `app/settings/selections/` 提供模型配置 UI。
- `app/lib/api/` 封装后端 REST/SSE API，`app/lib/api/types.ts` 维护共享类型。
- `app/hooks/use-chat-stream.ts` 负责流式聊天状态和 SSE 解析。

选用原因：

- App Router 适合把页面、加载态、错误边界和布局组织在同一目录树中。
- React 19 与 Next 16 是当前前端项目实际依赖版本。
- TypeScript strict 能让后端 API 变化更早暴露到前端调用层。

---

### 3.13 前端样式与 UI：Tailwind CSS v4 + 本地组件

前端样式使用 Tailwind CSS v4，主题 token 定义在 `frontend/app/globals.css`。

当前状态：

- 基础 UI 组件位于 `app/components/ui/`，包括 Button、Card、Badge、Toast、Drawer、Dialog、Spinner 等。
- 业务组件按 `chat/`、`resumes/`、`settings/` 分组。
- 页面风格偏后台工具：紧凑标题、列表卡片、清晰 loading/error/empty 状态。
- Markdown 输出使用 `react-markdown` 和 `remark-gfm`。

选用原因：

- Tailwind v4 适合快速维护一致的工具型界面。
- 本地 UI 组件避免过早引入重量级组件库。
- Markdown 渲染满足 AI 对话内容展示需要。

---

### 3.14 桌面壳：Electron 30 + Vite

Electron 项目位于 `electron/`，使用 Electron `30`、Vite `5`、TypeScript、React 18 和 Electron Builder。

当前状态：

- `electron/main.ts` 负责单实例、启动托管进程、日志、窗口生命周期和打包运行时配置。
- `electron/preload.ts` 暴露最小 bridge：`window.offerPilotRuntime.apiBaseUrl`。
- 开发模式查找相邻 `../backend` 和 `../frontend`，也支持 `OFFER_PILOT_BACKEND_DIR` 与 `OFFER_PILOT_FRONTEND_DIR`。
- 开发模式启动后端 `uv run uvicorn main:app`，启动前端 `npm run dev`。
- 生产模式从 `resources/backend/offer-pilot-api` 启动后端可执行文件，从 `resources/frontend/server.js` 启动 Next standalone server。
- 保持 `contextIsolation: true` 和 `nodeIntegration: false`。

选用原因：

- Electron 能把本地后端、前端和桌面窗口整合成一个用户入口。
- Vite 简化 Electron 开发期构建和主进程编译。
- preload 最小化有利于减少桌面端安全暴露面。

---

### 3.15 打包体系：Next standalone + PyInstaller + Electron Builder

桌面打包入口位于 `electron/package.json`。

当前状态：

- `npm run build:frontend` 会运行 `../frontend` 的 Next build，并用 `scripts/prepare-frontend.mjs` 把 standalone 输出复制到 `electron/resources/frontend`。
- `npm run build:backend` 会通过 `uv run --project ../backend pyinstaller ...` 打包后端，并用 `scripts/prepare-backend.mjs` staged 到 `electron/resources/backend/offer-pilot-api`。
- `npm run build:electron` 运行 `tsc && vite build`。
- `npm run build` / `npm run build:win` 生成 Windows x64 NSIS 安装包。

选用原因：

- Next standalone 适合把前端服务端运行依赖收敛到可复制目录。
- PyInstaller 适合把 Python 后端打包成桌面应用可托管的本地可执行文件。
- Electron Builder 提供 Windows 安装包生产链路。

---

### 3.16 日志与基础工具

后端日志通过 `loguru` 封装在 `backend/utils/logger.py` 中；通用流事件处理位于 `backend/utils/stream.py`。Electron 运行时会把托管进程日志写到 Electron `userData/logs`。

选用原因：

- `loguru` 比标准库 logging 更快落地。
- 流事件工具为 LangChain/LangGraph 事件处理提供公共抽象。
- 桌面端本地日志便于排查后端或前端子进程启动失败。

---

### 3.17 测试与质量检查

后端测试框架是 pytest，测试集中在 `backend/tests/unit/`。

当前后端覆盖范围包括：

- 配置解析。
- 数据库 URL、连接检查、建表和 PostgreSQL 可选集成。
- LangGraph 数据库 checkpointer。
- ORM、Repository、Service 层。
- 简历文件解析、预览和 API。
- 模型配置 API 和 OpenAPI 文档。
- AI 同步/流式接口、SSE 工具事件、interrupt/retry。
- Agent 图中的工具调用、并发、错误隔离、模型重试和异常逻辑。
- 模型装载错误封装。

前端和 Electron 当前没有专门测试 runner，主要依赖：

- 前端 `npm run lint` 和 `npm run build`。
- Electron `npm run lint`、`npm run build:electron` 和完整打包命令。

选用原因：

- pytest 与后端技术栈契合，适合验证 Agent、数据库兼容性和接口文档。
- 前端/Electron 当前优先用 lint 和 build 保证类型、路由和打包链路。

## 4. 已预留但未完全落地

`backend/agent/agents/resume_extractor/`、`backend/agent/agents/resume_advice/`、`backend/agent/agents/interview_prepare/` 和 `backend/agent/workflows/resume_extract/` 已存在，但当前对外 API 使用的是 `SupervisorAgent`。

结论：

- Agent 基础图、模型调用、工具调用和 checkpoint 已落地。

### 4.2 RAG 与向量检索

当前已经有 Web 搜索工具能力，但没有落地独立向量库、Embedding 存储或长期知识库检索链路。

结论：

- 外部信息获取能力已通过工具预留。
- ChromaDB、向量索引和完整 RAG pipeline 尚未引入。

---

### 4.3 前端自动化测试

前端目前有 lint 和 build，但没有引入 Vitest、Playwright 或 React Testing Library 测试体系。

结论：

- 现阶段通过类型检查、ESLint 和手动联调保障前端质量。
- 后续如果聊天流、简历预览和设置表单继续复杂化，需要补充组件测试或端到端测试。

---

### 4.4 数据库迁移体系

当前启动时通过 ORM metadata 自动建表，并维护 `sql/tables.sql` 结构快照，没有引入 Alembic。

结论：

- 本地开发和桌面应用原型阶段足够简单。
- 一旦需要长期升级已有用户数据库，应引入迁移版本管理。

## 6. 当前架构结论

截至目前，OfferPilot 的实际技术底座可以概括为：

- 单仓库三子项目：`backend/`、`frontend/`、`electron/`。
- Python 3.13 + uv。
- FastAPI + Pydantic v2 + ruamel.yaml。
- SQLAlchemy 2.x。
- SQLite 默认 + PostgreSQL 可选。
- LangChain + LangGraph。
- 数据库持久化 checkpoint。
- Exa Web Search 工具 + MCP 预留。
- PyMuPDF + Aspose.Words + RapidOCR ONNX Runtime。
- Next.js 16 App Router + React 19 + TypeScript strict。
- Tailwind CSS v4 + 本地 UI 组件。
- Electron 30 + Vite + Electron Builder。
