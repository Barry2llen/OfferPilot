# 求职助手 Agent 项目技术选型文档（V1）

## 1. 项目定位

本项目是一个面向求职场景的 Agent Web 应用，核心能力包括：

- JD 分析
- 简历优化建议
- 模拟面试
- 投递记录管理
- RAG 检索增强
- 多步骤任务编排

第一版目标：

- 聚焦 **Python + Agent + RAG**
- 支持真实场景使用
- 尽量做到 **开箱即用**
- 为后续扩展预留清晰的架构边界

---

## 3. 核心选型说明

## 3.1 Web 应用

项目第一版采用 Web 应用形态，方便开发、调试、展示与后续部署。

---

## 3.2 FastAPI

FastAPI 作为后端框架，负责：

- API 接口
- 文件上传
- 任务提交
- 结果返回
- 业务数据管理
- Agent 工作流接入

选用 FastAPI 的核心原因是：与 Python Agent 生态结合自然，开发效率高，适合快速完成 MVP。

---

## 3.3 LangChain + LangGraph

本项目的 Agent 能力与工作流编排明确采用：

- **LangChain**：用于模型调用封装、提示词链路、工具接入、输出解析
- **LangGraph**：用于定义多步骤工作流、节点状态流转与流程编排

在本项目中，它们主要承担以下职责：

### LangChain
- Prompt 模板管理
- LLM 调用封装
- 文档处理链路
- RAG 检索链路封装
- 工具调用能力集成

### LangGraph
- 任务流程编排
- 多节点执行控制
- 状态管理
- 条件分支
- 工作流可扩展设计

第一版的核心流程会围绕以下任务展开：

1. 接收用户输入（JD、简历、目标岗位）
2. 进入工作流
3. 执行文本分析与检索
4. 生成结构化结果
5. 保存结果并返回前端展示

---

## 3.4 SQLite

第一版业务数据存储采用 SQLite，用于保存：

- 用户基础信息
- 简历内容
- JD 数据
- 分析结果
- 投递记录
- 历史任务记录

选用 SQLite 的目标是降低启动门槛，便于本地运行和开源演示。

---

## 3.5 ChromaDB

第一版向量存储采用 ChromaDB，用于：

- 文档切分后的向量存储
- 简历/JD/知识库内容检索
- RAG 检索增强

ChromaDB 的定位是第一版默认向量数据库，优先满足：

- 本地可运行
- 接入简单
- 便于快速完成 RAG 能力

---

## 4. RAG 架构设计原则

RAG 是本项目的重要组成部分，第一版采用如下思路：

### 数据流
1. 上传原始文档
2. 文档切分
3. 向量化
4. 写入 ChromaDB
5. 查询时召回相关片段
6. 将检索结果交给大模型生成最终回答

### 第一版适用场景
- 简历内容检索
- JD 关键信息提取辅助
- 面试知识库检索
- 项目材料/个人经历增强生成

### 设计要求
- 检索流程和业务流程分层
- 向量存储与上层调用解耦
- 后续可扩展其他向量数据库实现
