# OfferPilot 技术选型说明

## 1. 文档目标

本文档描述的是仓库当前阶段的实际技术选型，而不是最初设想中的完整产品蓝图。判断标准分为三类：

- 已落地：仓库里已经有代码和测试支撑。
- 已预留：目录、依赖或抽象已经存在，但能力尚未完整上线。
- 暂未引入：早期方案里提过，但当前代码库并未采用。

---

## 2. 当前项目定位

OfferPilot 目前是一个以 Python 为核心的求职助手后端原型，重点在三件事：

- 建立可运行的 FastAPI 服务骨架
- 打通 Agent 模型调用与工具调用闭环
- 提供可切换模型提供商的配置、存储与服务层能力

现阶段已经实现的重点不是完整 Web 产品，而是后端基础设施和 Agent 执行底座。

---

## 3. 已落地选型

### 3.1 运行时与包管理

- Python `>=3.13`
- `uv` 作为依赖与命令执行工具

选用原因：

- Python 生态与 LangChain / LangGraph / FastAPI 兼容最好。
- `uv` 启动和依赖解析速度快，适合本地迭代和 CI 使用。

---

### 3.2 Web 框架：FastAPI

当前入口位于 `main.py`，已经使用 FastAPI 创建应用，并通过 lifespan 完成数据库初始化与释放。

当前状态：

- 已落地应用入口
- 已接入生命周期管理
- 当前仅保留基础路由，正式业务路由仍在补充
- `api/` 目录已预留，但尚未形成稳定接口层

选用原因：

- 与 Python Agent 生态衔接自然
- 异步支持和类型提示体验较好
- 适合作为后续文件上传、任务提交、结果查询的 API 容器

---

### 3.3 配置系统：Pydantic v2 + ruamel.yaml

当前配置由 `schemas/config/` 负责，`load_config()` 从 `config.yaml` 读取并解析为强类型对象。

当前状态：

- 已支持 `database` 根级配置
- 已兼容旧版 `offer_pilot.database` 结构
- 已支持 SQLite 与 PostgreSQL 两种数据库配置
- 已提供 `config.example.yaml`

选用原因：

- Pydantic v2 适合做强类型配置建模
- `ruamel.yaml` 保留 YAML 结构能力较好，便于后续扩展和示例维护

---

### 3.4 数据库层：SQLAlchemy 2.x

当前数据库访问基于 SQLAlchemy 2.x ORM，而不是早期文档中提到的 SQLModel。

当前状态：

- `db/engine/manager.py` 提供惰性初始化、会话工厂、连接检查与资源释放
- `db/models/` 已定义 `tb_chat`、`tb_model_provider`、`tb_model_selection`
- `db/repositories/` 已提供模型提供商与模型选择的数据访问封装
- 启动时通过 ORM metadata 自动建表

选用原因：

- SQLAlchemy 2.x 表达力和可控性更强，适合做基础设施层
- 当前模型数量不大，但已经需要清晰的 ORM、Repository、Service 分层
- 对 SQLite 与 PostgreSQL 双兼容更直接

---

### 3.5 数据库类型：SQLite 默认，PostgreSQL 可选

当前默认数据库是 SQLite，配置路径为 `./data/offer_pilot.db`；同时保留 PostgreSQL 配置模型与可选测试。

当前状态：

- SQLite 为默认运行模式
- PostgreSQL 配置、URL 构建与连接检查已实现
- PostgreSQL 集成测试依赖 `TEST_POSTGRES_*` 环境变量

这样选的原因：

- SQLite 启动门槛低，适合本地开发与原型验证
- PostgreSQL 作为后续部署或多环境运行的扩展方向，避免现在就把数据层锁死在单一实现

---

### 3.6 Agent 编排：LangChain + LangGraph

当前 Agent 基础能力已经落在 `agent/` 目录，核心图实现是 `agent/graphs/model_call.py` 中的 `ModelCallGraph`。

当前状态：

- 已支持模型调用节点
- 已支持工具调用节点
- 已支持模型输出 `tool_calls` 后回流工具执行
- 已支持模型调用失败后的重试与中断机制
- Agent 基础状态由 `TypedDict` 定义

LangChain 当前承担：

- 统一聊天模型初始化
- 工具绑定
- 消息对象与工具调用协议复用

LangGraph 当前承担：

- 状态图定义
- `model -> tool -> model` 闭环编排
- 条件分支与结束控制

选用原因：

- 当前项目不是简单的单次 LLM 调用，而是明显需要保留工作流和工具调用扩展点
- LangGraph 比手写 while-loop 更适合后续增加分支、人工中断和多节点状态

---

### 3.7 模型接入抽象：多提供商统一封装

当前已经落地“模型提供商 + 模型选择”两层抽象：

- `ModelProvider`：描述供应商、名称、`base_url`、`api_key`
- `ModelSelection`：描述实际使用的模型

已支持的提供商映射：

- OpenAI
- Anthropic
- Google
- OpenAI Compatible

选用原因：

- 避免把具体模型厂商硬编码在业务逻辑里
- 便于后续支持不同工作流绑定不同模型
- 兼容官方接口和自建兼容接口

---

### 3.8 日志与基础工具

当前日志通过 `loguru` 封装在 `utils/logger.py` 中。

选用原因：

- 比标准库 logging 更快落地
- 输出格式和调用体验更适合当前阶段的调试需求

---

### 3.9 测试框架：pytest

当前测试集中在 `tests/unit/`，覆盖范围包括：

- 配置解析
- 数据库引擎与建表
- ORM 模型行为
- Repository / Service 层
- Agent 图中的工具调用与重试逻辑
- 用户输入校验

选用原因：

- 与当前 Python 技术栈契合
- 便于围绕 Agent 节点和配置兼容性写细粒度测试

---

---

`api/` 目录已经创建，但正式路由、依赖注入、请求响应 schema 还没有完整展开。

结论：

- 方向明确
- 仍属于骨架阶段

---

以下内容出现在旧版文档中，但当前仓库并未真正采用：

---

## 6. 当前架构结论

截至目前，OfferPilot 的实际技术底座可以概括为：

- Python 3.13
- uv
- FastAPI
- Pydantic v2
- ruamel.yaml
- SQLAlchemy 2.x
- SQLite 默认 + PostgreSQL 可选
- LangChain
- LangGraph
- Loguru
- Pytest

项目当前最清晰的方向不是“完整求职平台”，而是“先把可扩展的 Agent 后端底座做扎实”，后续再逐步补齐简历解析、正式 API、RAG 和前端界面。
